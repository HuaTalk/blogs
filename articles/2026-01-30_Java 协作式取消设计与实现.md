## 背景

在高并发系统中，我们经常需要并行执行大量子任务。然而，当出现以下情况时，继续执行剩余任务往往是资源浪费：

1.  **超时**：总体任务已超时，继续执行无意义
2.  **快速失败（Fail-Fast）**：某个子任务异常，整批任务结果已不可用
3.  **主动取消**：业务逻辑判断不再需要结果

Java 原生的 `Thread.interrupt()` 只能中断阻塞操作（如 `sleep`、`wait`、`BlockingQueue.take`），无法中断正在执行 CPU 密集型代码的线程。因此，我们需要设计一套**协作式取消机制（Cooperative Cancellation）**。

## 设计目标

1.  **统一的取消信号传递**：所有并行子任务共享同一个取消令牌
2.  **支持多种取消原因**：超时、异常、主动取消
3.  **轻量级检查点**：任务可以在关键节点主动检查取消状态
4.  **延迟绑定**：取消令牌在任务提交后才绑定实际的 Future
5.  **资源自动释放**：取消后自动清理未完成任务

## 核心组件设计

### 1. CancellationTokenState - 取消状态枚举

首先定义清晰的状态枚举，避免魔法数字：

```java
public enum CancellationTokenState {
    /** 任务正在运行 */
    RUNNING(0),
    /** 任务成功完成 */
    SUCCESS(1),
    /** 空操作 */
    NO_OP(2),
    /** 因其他任务异常而取消（快速失败） */
    FAIL_FAST_CANCELLED(-1),
    /** 因超时而取消 */
    TIMEOUT_CANCELLED(-2),
    /** 因主动调用 cancel() 而取消 */
    MUTUAL_CANCELLED(-3);

    private final int value;

    /**
     * 判断是否需要取消子任务
     * 负值状态表示需要取消
     */
    public boolean shouldCancelSubtasks() {
        return this.value < 0;
    }
}
```

**设计要点**：

*   使用正值表示正常状态，负值表示取消状态
*   `shouldCancelSubtasks()` 方法提供语义化的判断接口

### 2. CancellationToken - 取消令牌

取消令牌是整个机制的核心，负责：

*   跟踪任务批次的整体状态
*   传递取消信号
*   支持延迟绑定

```java
@NoArgsConstructor
public class CancellationToken {
    // 使用 SettableFuture 支持延迟绑定
    private final SettableFuture<Object> futureToken = SettableFuture.create();
    // 原子状态，保证线程安全
    private final AtomicReference<CancellationTokenState> state =
        new AtomicReference<>(RUNNING);

    /**
     * 延迟绑定：在任务提交后绑定实际的 Future
     *
     * @param futures 所有子任务的 Future 列表
     * @param timeout 超时时间
     * @param submitCanceller 提交器的取消 Future
     */
    @SuppressWarnings({"unchecked", "raw"})
    public <T> void lateBind(List<ListenableFuture<T>> futures,
                             Duration timeout,
                             ListenableFuture<?> submitCanceller) {
        // 快速失败 Future：任何一个任务失败都会触发
        FluentFuture<?> failFastFuture = FluentFuture.from(Futures.allAsList(futures))
                .catchingAsync(Throwable.class,
                    ex -> Futures.immediateCancelledFuture(), directExecutor())
                .withTimeout(timeout, TimerService.getTimer());

        // 所有任务的组合 Future
        ListenableFuture<?> allFutures = Futures.successfulAsList(
            Futures.successfulAsList(futures), submitCanceller);

        // 监听快速失败 Future 的完成事件
        failFastFuture.addCallback(new FutureCallback() {
            @Override
            public void onSuccess(Object result) {
                state.compareAndSet(RUNNING, SUCCESS);
            }

            @Override
            public void onFailure(Throwable t) {
                // 取消所有任务
                allFutures.cancel(true);
                // 根据异常类型设置状态
                if (t instanceof TimeoutException) {
                    state.compareAndSet(RUNNING, TIMEOUT_CANCELLED);
                } else {
                    state.compareAndSet(RUNNING, FAIL_FAST_CANCELLED);
                }
            }
        }, directExecutor());

        // 绑定到 futureToken
        futureToken.setFuture(failFastFuture);
    }

    /**
     * 主动取消
     */
    public void cancel(boolean useInterrupt) {
        state.compareAndSet(RUNNING, MUTUAL_CANCELLED);
        futureToken.cancel(useInterrupt);
    }

    public CancellationTokenState checkState() {
        return state.get();
    }
}
```

**关键设计**：

1.  **延迟绑定（Late Binding）**：`SettableFuture` 允许先创建 Token，后绑定实际 Future，解决"先有鸡还是先有蛋"问题

2.  **快速失败语义**：使用 `Futures.allAsList()` + `catchingAsync` 实现——只要有一个任务失败，立即触发取消

3.  **超时控制**：`withTimeout()` 设置超时，超时后触发 `TimeoutException`

### 3. Checkpoints - 检查点机制

由于无法强制中断 CPU 密集型代码，我们在任务代码中插入"检查点"：

```java
public class Checkpoints {

    /**
     * 标准检查点：检查 cancelToken 状态
     * 如果任务已取消，抛出取消异常
     */
    public static void checkpoint(String taskName, boolean lean) {
        ParallelOptions options = TaskScopeTl.getParallelOptions();
        if (taskName == null || options == null
                || "task".equals(options.getTaskName())
                || !taskName.equals(options.getTaskName())) {
            return;
        }

        CancellationToken cancelToken = TaskScopeTl.getCancellationToken();
        if (cancelToken != null) {
            if (cancelToken.checkState().shouldCancelSubtasks()) {
                throw lean
                    ? new LeanCancellationException("Cancel during running")
                    : new FatCancellationException("Cancel during running");
            }
        }
    }

    /**
     * 原始检查点：只检查线程中断标志
     * 适用于不使用 cancelToken 的场景
     */
    public static void rawCheckpoint() {
        if (Thread.interrupted()) {
            throw new FatCancellationException("Cancel by interruption");
        }
    }
}
```

**两种取消异常**：

*   `FatCancellationException`：带完整堆栈，便于调试
*   `LeanCancellationException`：无堆栈，生产环境减少开销

### 4. ScopedCallable - 带作用域的 Callable 包装器

包装原始任务，添加上下文绑定和监控能力：

```java
public class ScopedCallable<V> implements Callable<V>, Attachable {

    public static final String KEY_PARALLEL_OPTIONS = "parallelOptions";
    public static final String KEY_CANCELLATION_TOKEN = "cancellationToken";

    private final String taskName;
    private final Callable<V> delegate;
    private final ConcurrentMap<String, Object> attachments = new ConcurrentHashMap<>();

    @Override
    public V call() throws Exception {
        // 1. 准备上下文：绑定 CancellationToken 到 ThreadLocal
        TaskScopeTl.init(getCancellationToken(), getParallelOptions());
        TaskPair.init(taskName);

        try {
            // 2. 执行检查点（任务开始前检查）
            Checkpoints.checkpoint(taskName, true);
            startTime = ticker.read();
            // 3. 执行实际任务
            return delegate.call();
        } finally {
            endTime = ticker.read();
            // 4. 清理上下文
            TaskScopeTl.remove();
            // 5. 记录监控指标
            recordMetrics();
        }
    }
}
```

**设计要点**：

*   通过 `Attachable` 接口支持任意上下文传递
*   任务执行前自动调用检查点
*   使用 `ThreadLocal` 传递取消令牌，子方法可随时检查

### 5. ConcurrentLimitExecutor - 并发限制执行器

控制同时执行的任务数量，实现"滑动窗口"提交：

```java
public class ConcurrentLimitExecutor<V> {

    private final ExecutorCompletionService<V> cs;
    private final BlockingQueue<ListenableFuture<V>> blockingQueue =
        new LinkedBlockingQueue<>();
    private final ParallelOptions options;

    /**
     * 提交所有任务，立即返回 Future 列表
     */
    public AsyncBatchResult<V> submitAll(List<? extends Callable<V>> tasks) {
        Builder<ListenableFuture<V>> resultBuilder =
            ImmutableList.builderWithExpectedSize(tasks.size());

        // 1. 立即提交初始批次（数量 = 并行度）
        int start = Math.min(tasks.size(), options.getParallelism());
        for (int i = 0; i < start; i++) {
            resultBuilder.add(fallbackSubmit(tasks, i));
        }

        int remaining = tasks.size() - start;
        if (remaining <= 0) {
            return AsyncBatchResult.<V>builder()
                    .results(resultBuilder.build())
                    .build();
        }

        // 2. 为剩余任务创建 SettableFuture 占位
        List<SettableFuture<V>> others = IntStream.range(0, remaining)
                .mapToObj(ignore -> SettableFuture.<V>create())
                .collect(toImmutableList());

        ImmutableList<ListenableFuture<V>> results =
            resultBuilder.addAll(others).build();

        // 3. 异步提交剩余任务（在独立线程中）
        ListenableFuture<?> submittingFuture =
            TimerService.getTimer().submit(() -> submitRemaining(tasks, results, start));

        return AsyncBatchResult.<V>builder()
                .submitCanceller(submittingFuture)
                .results(results)
                .build();
    }

    /**
     * 提交剩余任务（在独立线程执行）
     * 等待一个完成 -> 提交下一个
     */
    private int submitRemaining(List<? extends Callable<V>> tasks,
                                List<ListenableFuture<V>> result,
                                int start) {
        int index = start;
        while (index < tasks.size()) {
            // 阻塞等待任何一个任务完成
            ListenableFuture<V> completed = blockingQueue.take();

            // 检查是否已取消
            if (completed.isCancelled() || result.get(index).isCancelled()
                    || Thread.interrupted()) {
                return submitted;
            }

            // 提交下一个任务，绑定到对应位置的 SettableFuture
            ((SettableFuture<V>) result.get(index))
                .setFuture(fallbackSubmit(tasks, index));
            index++;
        }
        return submitted;
    }
}
```

**滑动窗口机制**：

1.  初始提交 N 个任务（N = 并行度）
2.  等待任何一个完成，释放一个"槽位"
3.  提交下一个任务
4.  重复直到所有任务提交完成

## 整体流程

下面是 `FlightParallelHelper.parMap` 的完整流程：

```java
public static <T, R> AsyncBatchResult<R> parMap(
        List<T> list,
        Function<? super T, ? extends R> function,
        ParallelOptions options,
        ListeningExecutorService executor,
        String executorName) {

    if (CollectionUtils.isEmpty(list)) {
        return emptyBatchResult();
    }

    // 1. 规范化参数
    ParallelOptions normalizedOptions = ParallelOptions.formalized(options, list.size());
    String taskName = normalizedOptions.getTaskName();

    // 2. 创建取消令牌（此时未绑定 Future）
    CancellationToken cancellationToken = CancellationToken.create();

    // 3. 包装任务，附加上下文
    List<Callable<R>> tasks = list.stream()
            .map(item -> {
                ScopedCallable<R> scopedCallable = new ScopedCallable<>(
                    taskName, () -> function.apply(item));
                scopedCallable.put(KEY_PARALLEL_OPTIONS, normalizedOptions);
                scopedCallable.put(KEY_CANCELLATION_TOKEN, cancellationToken);
                return (Callable<R>) scopedCallable;
            })
            .collect(toImmutableList());

    // 4. 通过限流执行器提交任务
    AsyncBatchResult<R> batchResult = ConcurrentLimitExecutor.<R>builder()
            .pool(executor)
            .options(normalizedOptions)
            .build()
            .submitAll(tasks);

    // 5. 延迟绑定：将 Future 列表绑定到 CancellationToken
    cancellationToken.lateBind(
        batchResult.getResults(),
        normalizedOptions.forTimeout(),
        batchResult.getSubmitCanceller());

    // 6. 超时时触发资源清理
    tryPurgeOnTimeout(executorName, batchResult);

    return batchResult;
}
```

## 在业务代码中使用检查点

对于 CPU 密集型任务，需要在关键位置插入检查点：

```java
public void processLargeDataSet(List<Data> dataList) {
    for (int i = 0; i < dataList.size(); i++) {
        // 每处理 100 条数据检查一次
        if (i % 100 == 0) {
            Checkpoints.checkpoint("processData", true);
        }

        processItem(dataList.get(i));
    }
}

public void complexCalculation() {
    // 阶段 1
    doPhase1();
    Checkpoints.checkpoint("complexCalc", true);

    // 阶段 2
    doPhase2();
    Checkpoints.checkpoint("complexCalc", true);

    // 阶段 3
    doPhase3();
}
```

## 总结

本文介绍的取消机制包含以下核心设计思想：

| 设计点        | 解决的问题                                 |
| ---------- | ------------------------------------- |
| **协作式取消**  | Java 无法强制中断 CPU 密集型代码                 |
| **延迟绑定**   | 解决"先创建 Token 还是先提交任务"的循环依赖            |
| **状态枚举**   | 清晰区分不同取消原因，便于监控和排查                    |
| **轻量级异常**  | 生产环境减少堆栈开销                            |
| **滑动窗口提交** | 控制并发度，避免线程池过载                         |
| **上下文传递**  | ThreadLocal + Attachable 实现跨方法的取消信号传递 |

这套机制已在生产环境稳定运行，有效减少了超时场景下的资源浪费，提升了系统整体吞吐能力。
